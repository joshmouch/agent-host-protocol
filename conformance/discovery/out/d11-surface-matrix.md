## 4. Cross-angle findings

### 4a. Spec-vs-schema gaps (from D2 × D1)

The schema inventory (D1, 131 rows) and the normative-prose inventory (D2, 77
rows) describe the surface in **two different vocabularies**, and the seam
between them is itself a finding:

- **Schematized but un-normed.** ~90 of D1's method/notification keys
  (`createSession`, `disposeSession`, `dispatchAction`, `fetchTurns`,
  `createTerminal`, `invokeChangesetOperation`, the whole `changeset/*` and
  `otlp/*` families, …) carry **no RFC-2119 clause** in `docs/specification/**`.
  They are typed in the schema but the spec never says MUST/SHOULD anything about
  them. These are the methods where a host could behave arbitrarily and still be
  "spec-compliant" — strong candidates for **spec-clarification PRs** *and* for
  conformance scenarios that pin the schema-implied contract.
- **Normed but off-schema-vocabulary.** D2's strong clauses cluster on *concept*
  nouns the schema doesn't name as methods — `versioning` (3 MUST),
  `UnsupportedProtocolVersion` (3 MUST), `channel` (3 MUST), `AuthRequired`,
  `error-response`, `reconnect-replay`, `notification`, `transport`. These are
  cross-cutting protocol rules (handshake, auth, error envelope, reconnect)
  rather than single RPCs, which is why they don't key 1:1 to a D1 method row.
- **Caveat (do not over-read):** the raw "29 D2 keys with no D1 element" /
  "90 D1 methods with no D2 prose" counts are inflated by this concept-vs-method
  keying mismatch (D2 rows are mostly `method:null`, concept-keyed; D1 rows are
  method-keyed). They are not 90 literal contradictions — they are a **vocabulary
  seam**. The actionable subset: the un-normed *core RPCs* above, which a Part-2
  scenario should pin and an upstream spec note should describe. All 30 D2
  MUST-class clauses are nonetheless individually mapped into the matrix (§2).

### 4b. Client divergences (from D8 — 16 confirmed)

All 16 D8 rows are `edge`-class and 11 carry an explicit `spec-gap` triage.
Dominant pattern: **unknown-variant / unknown-field preservation on round-trip**
— the 6 SDKs each implement an `Unknown`/raw fallback for unrecognized
discriminators, but the *representation differs* (Rust `serde untagged`, Go
`Raw []byte` + custom marshal, Kotlin `JsonObject` + KSerializer, .NET
`JsonElement`, Swift `AnyCodable`, TS compile-time-only with no runtime decoder).
Affected concepts: `StateAction` unknown variant, `SessionStatus` unknown bits,
`Customization` / `ResponsePart` / `ToolCallState` unknown kind/status,
`SessionSummary` unknown-key handling and optional-null encode omission.

- **`spec-gap` (11):** spec is silent on whether unknown variants MUST be
  preserved verbatim on re-encode. Corpus fixture `002` tests it but only for a
  subset of clients. **Each must become a pinned round-trip scenario, and the
  preservation rule should be proposed to the spec.**
- **`client-bug` (2):** `rpc.session-subscribe.edge.channel-scoped-notification-uri`
  and `action.SessionQueuedMessagesReordered.edge.leftover-order-stable` — one
  implementation diverges from the others; the majority behavior should be pinned
  and the outlier filed.
- **Untriaged (3):** `SessionToolCallReady` delta/invocation-message update,
  `SessionConfigChanged` replace-false merge semantics, `SessionInputRequested`
  upsert-answer preservation — the per-client runners must execute the differential run to
  resolve these to spec-gap vs bug.

### 4c. Mutation targets (from D9 — 66 surviving mutants)

66 mutants survived against the reference reducer + light-host protocol logic =
66 behaviors **no current scenario catches**. Operator distribution (where
tagged): `remove-conditional` 7, `replace-value` 5, `remove-branch` 5,
`negate-conditional`/`replace-enum-value`/`replace-operator`/`remove-call` etc.
1–2 each (35 carry no explicit operator tag in notes). By class: 41 `happy`, 23
`edge`, 2 `error` — i.e. most surviving mutants are on *happy-path reducer state
assignments* (e.g. `session.lifecycle.happy.creating-to-ready`: flipping
`SessionLifecycle.Ready` → `CreationFailed` goes uncaught), confirming the
current single happy-path seed barely constrains the reducer. These are the
concrete "add a scenario that asserts this exact state field" targets; the mutation run will
re-mutate after Part 2 and ratchet the kill-rate floor.
